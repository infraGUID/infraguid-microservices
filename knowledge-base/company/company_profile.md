# 18. Operating Model Details

InfraGuid operates with a blended consulting and managed services model. Consulting teams design and implement cloud platforms, while managed services teams keep those platforms healthy after go-live. Every client engagement is assigned an accountable delivery lead, a solution architect, and named engineering owners for infrastructure, security, automation, and observability. The delivery lead owns commercial coordination and delivery cadence. The solution architect owns technical quality, design decisions, and alignment with approved reference architectures. Engineering owners own implementation detail, peer review, evidence capture, and operational readiness.

The company separates design authority from delivery execution. Architecture decisions are reviewed through the architecture review board before production implementation. Delivery teams may recommend exceptions, but the exception must document the business need, risk, compensating control, owner, and expiry date. This prevents temporary shortcuts from becoming permanent operating patterns. Exceptions that affect IAM, network exposure, encryption, logging, client data, or production availability require security review before implementation.

All client work is tracked through a standard delivery system. Work items include acceptance criteria, environment scope, impacted services, rollback approach, and required approvals. Infrastructure changes must reference the associated ticket in the pull request and deployment record. Production changes must be traceable from request to approval, implementation, validation, and closure. This traceability is mandatory for audits and for post-incident review.

# 19. Engineering Quality Principles

InfraGuid engineers are expected to design for failure, automate repeatable work, and make operational behavior observable. A platform is not considered production-ready until it has health checks, logging, metrics, alerting, runbooks, backup or recovery procedures, and documented ownership. Designs must account for steady-state operation and failure modes such as AZ outage, dependency throttling, DNS drift, certificate expiry, deployment rollback, and capacity exhaustion.

Automation is preferred over manual action when the action is repeated, high risk, or required during incidents. Terraform is the standard for cloud infrastructure provisioning. GitHub Actions is the standard delivery mechanism unless a client mandates an approved alternative. Manual console changes are prohibited in managed environments except during emergency mitigation, and any emergency change must be captured, reviewed, and reconciled back into code after the incident.

Security is embedded in engineering practice. Engineers must avoid broad IAM permissions, public network exposure, unencrypted storage, hardcoded secrets, and unmanaged credentials. Every service must use least privilege IAM, encrypted data stores, centralized logging, and tagged resources. Security findings are tracked as engineering work, not as separate advisory notes. Critical findings block production release unless the Chief Technology Officer and Security Lead approve a documented exception.

# 20. Client Communication Standards

Client communication follows a predictable operating rhythm. Active projects use weekly status updates, milestone reviews, risk registers, and decision logs. Managed services clients receive operational reports covering availability, incidents, changes, security findings, cost trends, and improvement recommendations. Communication must be factual, concise, and supported by evidence such as monitoring screenshots, ticket references, deployment records, or Terraform plan summaries.

Incident communication uses severity-based templates. For SEV-1 incidents, the client receives an initial acknowledgement within the contractual response window, status updates at agreed intervals, and a post-incident summary after service restoration. Engineers must separate confirmed facts from hypotheses. If root cause is unknown, the communication should state the current impact, mitigation actions in progress, next update time, and accountable incident commander.

Architecture communication must be understandable to technical and non-technical stakeholders. Solution diagrams include trust boundaries, internet exposure, data stores, shared services, and operational dependencies. Design documents describe tradeoffs, alternatives considered, expected monthly cost, security controls, reliability assumptions, and rollback implications. Recommendations are written as actionable decisions rather than generic cloud advice.

# 21. Internal Review Boards

The Architecture Review Board evaluates production architecture proposals, major migrations, shared platform changes, and exceptions from standard patterns. Review criteria include alignment with reference architectures, network segmentation, IAM boundaries, observability coverage, disaster recovery approach, cost profile, and operational maintainability. Approval is recorded in the project decision log and linked to implementation pull requests.

The Security Review Board evaluates IAM policies, public access, encryption, secret handling, WAF rules, vulnerability findings, and compliance obligations. Security review is required for any system that processes regulated data, exposes an internet-facing endpoint, introduces a new identity provider integration, or changes production access controls. Security sign-off must include residual risks and control owners.

The Operations Review Board evaluates production readiness before go-live. Required evidence includes deployment runbook, rollback runbook, monitoring dashboards, alert routes, backup configuration, incident contacts, access model, support handover, and known limitations. A platform may not enter managed services until operations accepts the handover package.

# 22. Managed Services Commitments

Managed services commitments are based on clear service boundaries. InfraGuid is responsible for infrastructure components, platform automation, monitoring configuration, incident response within scope, and agreed operational reporting. Application teams remain responsible for application code, business logic, content, and client-owned data unless the statement of work explicitly assigns those responsibilities to InfraGuid.

Operational changes are governed by maintenance windows and change categories. Standard changes are pre-approved, repeatable, low-risk tasks with documented rollback. Normal changes require approval and scheduling. Emergency changes are allowed only to restore service, reduce active risk, or meet a time-sensitive security requirement. Every emergency change receives retrospective review.

Service health is measured through client-approved indicators. Common indicators include endpoint availability, ALB target health, RDS CPU and storage, queue depth, error rate, deployment success rate, backup success, patch compliance, and security finding age. Monthly reports highlight trends rather than only point-in-time metrics, allowing teams to identify capacity, reliability, and cost issues before they become incidents.

# 23. Knowledge Management

InfraGuid treats operational knowledge as a managed asset. Runbooks, standards, incident history, architecture decisions, and Terraform module documentation must be kept current. Documents have owners, review cycles, classifications, and related-document links. When an incident reveals a documentation gap, the corrective action must include updating the relevant runbook or standard.

Knowledge base content is written for retrieval and operational use. Documents use descriptive headings, concrete procedures, examples, commands, expected outcomes, and escalation criteria. Engineers avoid vague statements such as "follow best practices" unless the document defines the actual approved practice. Search terms and metadata are maintained so internal tools can retrieve the right guidance during time-sensitive work.

Lessons learned are shared through engineering forums and internal enablement sessions. Major incidents, new reference architectures, security policy changes, and Terraform module updates are presented to affected teams. The goal is to reduce repeated mistakes and make approved patterns easier to apply than ad hoc alternatives.
